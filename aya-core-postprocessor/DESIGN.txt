CO-RE Post-Processor Design
============================

Goal
----
Make the `core_read!` macro portable across kernel versions by adding
BPF CO-RE (Compile Once - Run Everywhere) relocation records to
compiled BPF ELF objects.  At load time, aya's existing `relocate_btf()`
patches field offsets to match the target kernel's struct layout.

Background: How CO-RE Relocations Work
---------------------------------------
A `.BTF.ext` section contains `bpf_core_relo` records.  Each record says:

    struct bpf_core_relo {
        u32 insn_off;          // byte offset of the instruction in its ELF section
        u32 type_id;           // BTF type id of the struct being accessed
        u32 access_str_off;    // offset into BTF string table: "0:3:1" etc.
        u32 kind;              // relocation kind, e.g. BPF_CORE_FIELD_BYTE_OFFSET (0)
    };

The `access_str_off` points to an "access string" like "0:2:1" which
encodes a chain of member indices.  For `task_struct.scx.dsq_vtime`,
if `scx` is the 3rd member (index 2) and `dsq_vtime` is the 2nd member
(index 1), the access string would be "0:2:1".

At load time, aya's `relocate_btf()`:
  1. Finds the local BTF type by `type_id`
  2. Parses the access string to build an `AccessSpec`
  3. Searches the target kernel's BTF for a matching struct
  4. Walks the target's members along the same named path
  5. Computes the new byte offset
  6. Patches the BPF instruction (LDX/STX offset field, or MOV immediate)

The Marker Mechanism
--------------------
The post-processor needs to find BPF instructions that encode a
compile-time field offset, and know which struct type and field path
they correspond to.  The marker must survive compilation through
rustc + LLVM's BPF backend.

Design: Custom ELF section with relocation descriptors.

The `core_read!` macro will emit a companion entry in a custom ELF
section called `.aya.core_relo`.  Each entry is a fixed-size record
containing:

    struct CoreReloMarker {
        section_name_hash: u64,   // FNV-1a hash of the ELF section name (e.g. "tracepoint/...")
        insn_byte_offset:  u64,   // byte offset of the LDX/MOV instruction within the section
        struct_name_len:   u32,   // length of struct name string
        field_path_len:    u32,   // length of field path string
        // followed by: struct_name bytes (NUL-terminated)
        // followed by: field_path bytes (NUL-terminated, dot-separated like "scx.dsq_vtime")
    }

However, for a prototype, we use a simpler approach:

Prototype Marker: inline asm with a `.pushsection` directive.

The macro emits inline assembly that:
  1. Writes a record into `.aya.core_relo` containing the struct name,
     field path, and a reference to the instruction being patched.
  2. The record uses a local label to capture the instruction offset.

This is modeled on how C BPF programs use `__attribute__((section(...)))`
to annotate relocatable accesses.

But even this is complex for a prototype.  The simplest viable approach:

Prototype Marker (v1): A map from instruction patterns to metadata.

For the initial prototype, we avoid modifying the macro.  Instead, the
post-processor works with a JSON/TOML sidecar file that lists:

    [[relocation]]
    section = "tracepoint/sched/sched_switch"
    insn_offset = 42            # instruction index (not byte offset)
    struct_name = "task_struct"
    field_path = "pid"          # or "scx.dsq_vtime"

The sidecar file can be generated by a build script that analyzes the
`core_read!` call sites via proc-macro metadata, or hand-written for
prototyping.

Actual prototype approach chosen: INLINE ASM MARKERS
----------------------------------------------------
The macro emits a `.pushsection` / `.popsection` inline asm block that
writes marker data into `.aya.core_relo`.  The post-processor reads
this section, cross-references it with the `.BTF` section, and
generates `bpf_core_relo` records in `.BTF.ext`.

For the prototype we implement the post-processor and an EXAMPLE sidecar
file showing the format.  The macro integration is a separate step.

Post-Processor Architecture
----------------------------
Input:  BPF ELF file + sidecar TOML file
Output: Modified BPF ELF file with CO-RE records added to .BTF.ext

Steps:
  1. Parse the ELF to extract `.BTF` and `.BTF.ext` sections
  2. Parse the BTF to build a type database
  3. Read the sidecar file listing (section_name, insn_index, struct_name, field_path)
  4. For each relocation entry:
     a. Find the BTF type_id for the named struct
     b. Walk the field path to build the access string (colon-separated member indices)
     c. Add the access string to the BTF string table
     d. Create a `bpf_core_relo` record
  5. Rebuild the `.BTF.ext` section with the new core_relo records appended
  6. Rebuild the `.BTF` section with any new strings
  7. Write the modified ELF

.BTF.ext Binary Layout
-----------------------
The `.BTF.ext` section has this layout:

    btf_ext_header {
        magic: u16,          // 0xEB9F
        version: u8,         // 1
        flags: u8,           // 0
        hdr_len: u32,        // size of this header (typically 32)
        func_info_off: u32,  // offset from end of header to func_info
        func_info_len: u32,  // length of func_info data
        line_info_off: u32,  // offset from end of header to line_info
        line_info_len: u32,  // length of line_info data
        core_relo_off: u32,  // offset from end of header to core_relo
        core_relo_len: u32,  // length of core_relo data
    }

Each info section (func_info, line_info, core_relo) starts with:
    rec_size: u32            // size of each record

Then groups of per-section records:
    sec_name_off: u32        // offset into BTF string table
    num_info: u32            // number of records for this section
    records[num_info]        // the actual records

For core_relo, each record is a `bpf_core_relo` (16 bytes):
    insn_off: u32            // byte offset of instruction in its section
    type_id: u32             // BTF type id
    access_str_off: u32      // BTF string offset for access string
    kind: u32                // BPF_CORE_FIELD_BYTE_OFFSET = 0

The rec_size for core_relo is sizeof(bpf_core_relo) = 16.

Access String Format
---------------------
For field access `task_struct.scx.dsq_vtime`:
  - Start with "0" (base struct, first element = index into array of that type)
  - ":N" for each member, where N is the member index in the struct

Example: if task_struct has members {..., scx (index 35), ...}
         and sched_ext_entity has members {..., dsq_vtime (index 7), ...}
         then access_str = "0:35:7"

The post-processor walks the BTF type info to compute these indices.
